ASR 噪声下的结构化抽取评测：从文本相似到实体归因

Posted on 2026-06-10 Edited on 2026-06-18

ASR 结果用于结构化抽取时，评测不能只看整句文本相似度。真实系统关心的是关键实体有没有抽对、位置是否合理、错误来自识别还是 NER、以及噪声会不会把下游结构化结果带偏。

因此，ASR + NER 的评测要从“文本像不像”转到“实体是否可归因”。

要解决的问题

同样的 WER 可能对应完全不同的下游影响。一个虚词错了可能无关紧要，一个地址、姓名、编号或时间错了会直接破坏结构化结果。

如果只看 ASR 文本 precision / recall 或整体 WER，无法解释实体错误来自哪里：是 ASR 没识别出来，还是 NER 没抽出来，还是文本规范化把实体改坏了。

最小抽象

评测样本至少要同时保留三层信息：

alignment 用来把实体 span 对齐到 ASR 输出；error_reason 用来区分漏识别、替换、插入、边界错误、实体类型错误和规范化错误。

工程闭环

指标要拆成三组。

ASR 层看 WER/CER、关键词召回、实体词召回。

NER 层看 entity precision、entity recall、span F1、type accuracy。

归因层看实体错误中有多少来自 ASR，有多少来自抽取模型，有多少来自后处理规则。

这样才能决定下一步该做什么：补噪声数据、加热词、改 NER 训练集、调整规则，还是优化文本规范化。

设计取舍

实体位置匹配有三种常见策略。第一种是绝对 span 匹配，要求预测实体在文本里的起止位置与标注完全一致。这种方式严格，但 ASR 只要前面多识别一个字，后面所有 span 都会错位。第二种是全文搜索，只要预测文本里出现同名实体就算命中。这种方式宽松，但如果同一个实体在远处重复出现，会把错误位置也算成正确。

第三种是基于字符对齐的局部窗口匹配：先把 reference 和 ASR 输出对齐，再在实体附近的小窗口内判断是否命中。它承认语音识别会带来位置漂移，但不允许模型从整段文本任意位置捡一个相同字符串来冒充正确实体。

大规模评测还需要候选过滤。可以用多模式匹配先找到可能实体，再进入对齐和归因阶段。这样既保留速度，也避免每个样本都跑重模型。

评测矩阵

一套实用矩阵可以按实体是否出现、实体是否落在正确局部窗口拆开。reference 有而 ASR 无，是识别漏召回；ASR 有而 NER 无，是抽取漏召回；ASR 有、NER 有但 span 偏移，是边界问题；ASR 有、NER 有但 type 错，是类型问题；reference 无而 prediction 有，则是插入或未标注实体，需要抽检确认。

最后一类很容易被误用。不是所有 prediction 多出来都一定是 false positive，因为标注集可能漏标。评测脚本应保留可复核样本，而不是把所有额外实体直接打成错误。

失败归因

结构化抽取最常见的陷阱是把 ASR 和 NER 的错误混在一起。ASR 把实体听错，NER 再稳也无能为力；ASR 正确但 NER 漏抽，应该改抽取模型；ASR 和 NER 都对，但后处理把格式改坏，应该修规则。

因此，评测报告不要只给一个 F1。要输出实体级错误样本、对齐片段、局部窗口、错误类型和可复核原因。这样指标下降才会变成工程任务，而不是一串无法行动的数字。

落地检查

做这类评测时，第一版脚本不要追求复杂模型，而要先把样本级可解释性做出来。每个错误实体旁边放 reference 片段、ASR 片段、预测片段、局部对齐窗口和错误类型。只要评测报告里看不到这些局部证据，precision 和 recall 再完整也很难指导修复。

还要把高频实体和长尾实体分开看。高频实体错误通常暴露规则或规范化问题，长尾实体错误更容易暴露声学识别、热词覆盖和标注稀疏问题。混在一起统计，会让模型看起来只是整体不稳，实际上不同实体桶的修复路径完全不同。

直接结论

ASR 噪声下的结构化抽取评测，关键不是把所有错误合成一个分数，而是把错误分解到 ASR、NER、span 对齐和后处理。只有错误可归因，系统才知道下一轮应该改模型、改数据还是改规则。
