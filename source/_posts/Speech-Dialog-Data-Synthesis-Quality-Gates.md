合成对话数据的风险不是“生成得不够多”，而是生成得太像模板、太干净、太不符合语音输入的真实噪声。数量扩大很容易，难的是让样本既覆盖任务，又不把模型训练成只会回答标准文本。

所以语音对话合成数据要先设计质量闸门，再谈规模。

<!--more-->

## 要解决的问题

对话数据要服务下游训练和评测。它既要有结构化 schema，保证字段、意图、实体和状态可控；又要有口语化表达，覆盖停顿、重复、自我修正、n-best 候选和 ASR 错误。

如果只生成标准书面语，模型上线后会被真实口语击穿。如果只追求随机扰动，样本又会失去任务结构。

## 最小抽象

一条合成样本至少要包含：

```text
dialogue_id
turns
intent
entities
state
n-best
quality_flags
reason code
```

`quality_flags` 记录口语化、实体覆盖、逻辑一致性、敏感词、重复和异常格式。`reason code` 说明样本为什么保留、丢弃或需要人工复核。

## 工程闭环

质量闸门可以分三层。

第一层是 schema 校验：字段是否完整、实体是否在合法范围、状态转移是否合理。

第二层是语言质量：是否过于模板化、是否包含自然口语、是否有重复和自我修正。

第三层是训练影响：加入这批数据后，目标任务指标是否提升，错误类型是否减少，是否引入系统性偏差。

每一批合成数据都应该能回溯到生成配置、过滤规则和评测结果。否则数据越多，问题越难定位。

## 设计取舍

语音对话合成有一个核心矛盾：schema 要硬，语言要软。schema 太硬，样本会变成模板填空，模型学到的是字段顺序而不是交互能力。语言太软，生成内容会偏离任务，实体、状态和轮次关系无法验证。

比较可控的做法是先生成结构化计划，再生成口语化表达，最后用校验器把表达重新映射回 schema。n-best 也不应该只是装饰字段。语音系统里的候选结果代表不确定性，训练样本可以保留同一轮的多个可能表达，让下游模型学会在模糊输入里维持状态，而不是只处理唯一正确文本。

负样本同样重要。只生成顺利完成的对话，会让模型遇到缺字段、冲突实体、重复确认和用户修正时没有处理经验。合成数据应该显式覆盖边界样本，并把边界原因写进 quality_flags。

## 评测矩阵

合成数据评测要同时看离线质量和训练收益。离线质量包括 schema 通过率、字段覆盖率、实体合法率、轮次一致性、口语化比例、重复率和敏感内容过滤率。训练收益包括目标任务指标、状态跟踪错误、实体抽取错误、异常输入恢复能力和对真实口语样本的泛化。

最关键的是增量评测。每次只加入一批合成数据，观察错误分布如何变化。如果总量一起加入，指标变差时无法判断是哪个生成模板、哪个口语化策略或哪个过滤规则导致的。

## 失败归因

合成数据的坏味道通常有四种。第一是模板味太重，模型学会了固定表达。第二是约束漂移，生成内容看起来自然但 schema 对不上。第三是边界样本不足，系统在修正、打断、重复和缺字段时不稳。第四是过滤只看文本表面，没有记录为什么通过或失败。

这些问题都不能靠再生成一批解决。需要把生成、校验、过滤、训练和评测串成闭环。
## 落地检查

第一批合成数据不要直接追求数量，可以先做一个小闭环：固定少量 schema，生成一批口语化样本，跑 schema 回填校验，再抽查失败样本。失败样本要保留原因：字段缺失、状态冲突、表达过于模板化、实体漂移、轮次不一致，或者只是过滤器过严。

随后再把这批样本加入训练做增量对照。只要增量评测没有记录错误类型变化，就不能证明合成数据真的有效。一个有用的合成流程，应该能回答“它改善了哪类错误，又引入了哪类新偏差”。

## 直接结论

语音对话合成数据的核心不是生成，而是质量治理。先把 schema、quality_flags、n-best、reason code 和评测回流建起来，再扩大规模。没有这些闸门，合成数据会把训练目标污染成系统性偏差。
